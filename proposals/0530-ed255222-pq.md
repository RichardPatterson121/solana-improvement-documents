## Summary

Introduce a new hybrid signer account type (`HybridSignerV1`) and accompanying
on-chain verification program to the Solana ecosystem, supporting both Ed25519
and ED255222-PQ (ML-DSA-44, NIST FIPS 204) signatures. This enables a phased,
zero-downtime migration from Ed25519 to post-quantum cryptography.

## Motivation

Ed25519 is vulnerable to Shor's algorithm on cryptographically relevant quantum
computers. INRIA (2026) estimates 1,193 logical qubits suffice to break 256-bit
elliptic curve groups, compressing the threat timeline. Solana's transaction
signing, validator identity, stake-weighted QoS, and DeFi protocol authorization
all rely on Ed25519. Without a migration path, Solana is structurally exposed.

The Solana Foundation and Anza have separately begun post-quantum planning
(Falcon, ML-DSA). ED255222-PQ provides:
- A named, versioned, RFC-style specification submitted to IETF CFRG.
- A hybrid account model with 3-phase migration.
- A reference Rust implementation (`ed255222-pq`) and test vectors.
- A Lean 4 formal specification with a correctness theorem.
- An on-chain Solana verification program (non-breaking, Phase 1).

## New Terminology

- **HybridSignerV1**: On-chain account holding Ed25519 + ED255222-PQ public keys.
- **ED255222-PQ**: Profile of ML-DSA-44 (FIPS 204) with Solana domain separation.
- **Hybrid Phase**: 1 = Hybrid (both sigs), 2 = PQ-Only, 3 = Legacy-Deprecated.

## Detailed Design

### Account Structure (HybridSignerV1, 1346 bytes)

```
version:     u8        = 1
ed25519_pk:  [u8; 32]
pq_pk:       [u8; 1312]
phase:       u8        (1, 2, or 3)
```

### Hybrid Signature Wire Format (2484 bytes)

```
ed25519_sig: [u8; 64]
pq_sig:      [u8; 2420]
```

### Verification Policy

| Phase | Required |
|-------|----------|
| 1 | Ed25519 AND ED255222-PQ |
| 2 | ED255222-PQ only |
| 3 | Ed25519 (legacy fallback) |

### Domain Separation

```
DST_KEYGEN = "ED255222-PQ-KEYGEN"
DST_SIGN   = "ED255222-PQ-SIGN"
Context strings (e.g., "SOLANA-TX", "IDENTITY", "UPGRADE")
```

### Program Instructions (Borsh)

```rust
CreateHybridSigner { ed25519_pk: [u8;32], pq_pk: [u8;1312] }
VerifyHybrid { message, ed25519_sig: [u8;64], pq_sig: [u8;2420], context }
UpgradeToPQOnly { auth_pq_sig: [u8;2420] }
```

## Alternatives Considered

1. **Falcon**: Smaller sigs (1,281 bytes); being evaluated by Anza.
   ML-DSA-44 is NIST FIPS 204 final standard — preferred for compliance.
2. **Ed448**: Still quantum-vulnerable.
3. **Pure ML-DSA (no hybrid)**: Breaks existing wallets immediately.
4. **No action**: Structurally unacceptable given quantum timeline.

## Impact

- **dApps / DeFi**: Use `VerifyHybrid` via CPI. SDK update required.
- **Validators**: No change in Phase 1. Phase 2 requires runtime SIMD.
- **Wallets**: Must generate and store ED255222-PQ keys alongside Ed25519.
- **Core contributors**: Phase 1 is program-only and non-breaking.
  Phase 2 adds a new native signer type (runtime change, requires SIMD vote).

## Security Considerations

- ML-DSA-44 security: NIST category 2 (MLWE + MSIS hardness).
- Hybrid EUF-CMA: Secure if either component is EUF-CMA secure.
- **CVE-2026-24850**: Pin `ml-dsa >= 0.1.1` (duplicate hint index bug).
- All verification MUST be constant-time.
- Side-channel analysis required before Phase 2 adoption.
- Independent audit of ED255222-PQ domain separation required.

## Drawbacks

- Large signature sizes: 2,484 bytes vs 64 bytes for Ed25519.
- Compute unit cost for on-chain ML-DSA-44 verification must be
  benchmarked and fee-scheduled before Phase 2.
- Wallet ecosystem requires coordinated upgrade.

## Backwards Compatibility

- Phase 1: Fully non-breaking (opt-in program-level feature).
- Phase 2: New signer type; Ed25519 not removed.
- Phase 3: Deprecation requires ecosystem-wide consensus and timeline.

## Open Questions

1. Should Phase 2 use a host function (syscall) for ML-DSA verification
   to reduce compute unit cost?
2. Should ED255222-PQ adopt Falcon parameters instead for smaller sigs?
3. What is the acceptable compute unit ceiling for on-chain PQ verification?
